# 项目拆解与论文路线助手

把用户提供的论文、文献列表、PDF、链接、现有笔记或总结材料，整理成可以持续追加的结构化文档。默认先做逐篇抽取，再做跨文献综合，避免只停留在摘要级复述。对于方法类论文，要尽量还原从原始数据到最终输出的完整技术链路，而不是只写结论。

## 适用任务

- 阅读单篇论文并输出结构化卡片
- 比较多篇论文并写出统一口径的对照表
- 将用户提供的已有笔记、总结、讲义或草稿整理成可复用文档
- 围绕某个主题生成文献综述、研究脉络、方法演进路线或参考文献清单
- 从一个完整科研项目出发，拆解研究流程、关键难题、候选对策、文献来源、论文边界和可发表路线
- 判断某个技术方案是适合独立成文、作为补充实验、汇入系统论文，还是迁移成通用方法论文
- 持续追加已有文档，而不是把信息散落成很多零碎笔记

只有当用户明确要求“帮我找论文”“搜索某个方向的文献”时，才进入检索流程。否则优先处理用户已经给出的论文、文件或链接。

## 主流程

### 1. 识别任务类型

先判断当前任务属于哪一类：

- 单篇阅读
- 多篇比较
- 多篇整理成综述草稿
- 明确要求的文献检索
- 项目级论文路线拆解

如果核心目标是“阅读、比较、整理已有文献”，不要自行扩展成搜索任务。

如果核心目标是“完整项目怎么拆论文”“这个项目有哪些难题和对策”“哪些模块能成文/能通用化”“形成一个知识图谱和论文路线”，进入 **项目级论文路线拆解流程**。此时先读 `references/project-roadmap-workflow.md`，再按任务需要读取其指向的参考文件。

### 2. 明确输入材料与输出目标

先确认以下信息：

- 输入材料是什么：PDF、网页链接、题目列表、已有笔记、Markdown、DOCX 等
- 用户想要什么：单篇卡片、多篇对照、综述草稿、研究笔记、方法演进总结
- 输出是否要写入一个可持续维护的统一文档

默认输出为一份持续追加的 Markdown 文档。若用户已经有文档结构，优先沿用原结构，并在其基础上补充，而不是拆成多个零散文件。

### 3. 逐篇抽取，再进入综合

先对每篇文献做统一抽取，再做跨文献综合，不要直接把多篇文章的摘要拼接在一起。

逐篇抽取时遵守以下原则：

- 只写论文明确报告的内容
- 推断必须和明确陈述区分开
- 缺失信息写“文中未明确报告”，不要猜测
- 与当前任务无关的信息尽量不保留
- 多篇论文之间尽量使用同一套字段，便于比较

### 4. 方法类论文要还原“方法学链路”

对于方法、模型、流程、系统、算法类论文，不能只写“用了某模型”或“提高了准确率”。要尽量写清楚：

- 原始数据是什么
- 实际输入表示是什么
- 中间做了哪些预处理、编码、过滤、特征提取或降维
- 模型结构是什么
- 最终输出是什么
- 结果如何被评价
- 这篇工作在时间线上处于什么位置
- 它相对前一代方法改进了什么、保留了什么限制

如果用户研究的是 mNGS、宏基因组、物种分类、taxonomic classification、k-mer 表示、降维或深度学习分类任务，额外优先关注：

- 任务层级：species / genus / family / kingdom / phylum
- 输入表示：vectorized k-mer、sequence token、raw sequence
- k 值与 canonical 设置
- 模型结构：CNN、RNN/LSTM/BiLSTM、Attention、Transformer、BERT-like、MLP
- 数据类型：真实 reads、模拟 reads、contigs、16S、shotgun
- 是否 hierarchical classification
- 是否讨论 calibration / confidence / confidence threshold / ECE
- 与 baseline 的差异
- 数据是否过于理想化，是否存在泄漏或过于容易的评估设置

## 文献综合时必须回答的问题

跨文献综合不能只是“罗列结论”，至少要回答下面几类问题中的大部分：

- 这一主题的研究问题是如何定义和演化的
- 不同论文在哪些方面形成共识
- 它们在哪些方法、数据、评估或解释上存在分歧
- 哪些结论依赖更强的数据、标签、模型或评估设定
- 当前研究空白在哪里
- 哪些后续工作是真正的方法升级，哪些只是数据更丰富或评估更宽松

## 项目级论文路线拆解

当用户给出的不是一批论文，而是一个正在开发的科研项目或方法体系时，不要直接写综述。先把项目转成可分析对象：

1. **流程树**：从原始输入到最终输出，画出完整技术/科学流程。
2. **难题树**：逐层标出每个环节的 failure mode，不只列用户已经想到的问题，也要补充遗漏问题。
3. **对策树**：为每个难题提出候选模块、实验闭环和失败条件。
4. **文献图谱**：按难题和对策查找/整理来源文献，区分领域文献与跨领域方法文献。
5. **论文边界矩阵**：判断每个对策能否独立成文，是否只是系统模块，是否能迁移为通用方法论文。
6. **任务面板**：把后续工作拆成可执行阶段，每阶段给出输入、任务、输出和 go/no-go 标准。

项目级拆解要优先使用中文解释方法名；英文术语只作为括注或候选投稿题名保留。

详细流程见：

- `references/project-roadmap-workflow.md`：完整项目拆解总流程
- `references/problem-tree.md`：难题树构建规则
- `references/solution-literature-map.md`：对策和文献图谱规则
- `references/paper-boundary-matrix.md`：论文边界和成文判断规则
- `references/task-panel.md`：阶段化任务面板模板

项目级拆解可以提醒调用其他 skill：

- 需要真实检索文献时，调用或遵循 `nature-academic-search`
- 需要写 introduction / method / experiments / discussion 时，调用或遵循 `nature-writing`
- 需要审稿人视角评估可发表性时，调用或遵循 `nature-reviewer`
- 需要管理数据可用性、代码可用性和仓库发布时，调用或遵循 `nature-data`

如果是方法论主题，还要把文献放回时间线上：

- 早期方法解决了什么问题
- 中期方法在哪一环做了升级
- 后期方法为何需要新的数据、表示、模型或训练策略
- 某种“更强”的方法是否依赖更好的数据、更多标签或更高算力

### 5. 结合发表时间整理方法演进

对于同一方向的多篇论文，不要把它们当作静态并列对象。应尽量整理出一条方法论进化路径，例如：

- 数据表示如何变化
- 模型结构如何变化
- 评价指标如何变化
- 任务设定如何变化
- 研究重点是从“能不能做”变成“能不能做得更稳、更快、更可解释”

如果用户关注的是方法创新可发表性，还要额外写：

- 这条路线目前最强的主流方案是什么
- 用户当前方案与主流方案差在哪
- 缺的不是单点精度，还是完整证据链
- 哪些实验是最低限度必须补齐的

### 6. 输出一份统一文档

默认输出为单一 Markdown 文档，结构建议如下：

```markdown
# 主题

## 任务范围

## 文献清单

## 逐篇文献卡片

## 方法演进与跨文献综合

## 证据质量与比较公平性

## 研究空白与争议

## 可直接复用的结论

## 参考文献
```

如果用户要的是更像正式综述的文档，综合部分建议进一步展开为：

```markdown
# 主题

## 研究背景与问题定义
## 文献纳入与分类原则
## 方法流派与时间演化
## 逐篇文献卡片
## 跨文献比较与综合
## 证据质量与公平性评估
## 研究空白与未来方向
## 参考文献
```

## 逐篇卡片字段

多数任务都应尽量按同一字段抽取，建议字段如下：

- 标题
- 作者
- 年份
- 发表 venue / 期刊 / 会议
- 研究问题
- 研究对象或任务
- 数据集 / 语料 / 实验对象
- 方法
- 基线 / 对照
- 评价指标
- 主要结果
- 作者声称的贡献
- 局限性
- 与当前主题的相关性
- 时间位置 / 所处发展阶段

如果论文是方法类、模型类或流程类，还要补充：

- 原始数据类型
- 模型实际输入数据
- raw data 到模型输入的转换步骤
- 方法类型
- 任务定义
- 标签或真值来源
- 评价方式与关键指标
- 最终输出
- 可解释性
- 适用边界
- 相对前一代方法的改进点
- 相对前一代方法保留的限制
- 在时间演化中的位置

如果论文和当前项目强相关，建议再补一行简评：

- 对我们当前工作的启发
- 对我们方法选择的支持或挑战
- 是否值得重点精读
- 对后续实验或写作是否有直接帮助

## 抽取规则

- 先写论文明确说了什么，再写基于内容的合理推断
- 任何无法确认的信息都要标注“文中未明确报告”
- 不要用猜测补齐字段
- 多篇论文尽量使用同一套字段，方便比较
- 如果存在明显的前后承继关系，要显式记录“前一代方法的问题”和“新方法的改进”
- 如果论文核心是方法演进，不要只总结最终精度，要把输入表示、模型结构和数据条件一起写清楚

## 文献综合规则

综合部分不要只是把卡片顺序拼起来。至少要写出以下内容中的多数：

- 研究问题是如何定义的
- 哪些论文属于同一流派
- 不同流派的核心假设是什么
- 数据和标签条件是否公平
- 方法演进是由数据、表示、模型还是评估驱动
- 结论是否在不同论文中一致
- 当前空白和未来方向是什么

如果用户的目标是判断一个项目能否发展成论文，综合部分还应额外输出：

- 当前方案的潜在创新点
- 与主流方法的差距
- 最小可发表实验集合
- 主要风险
- 需要补的证据链

## 检索流程

只有当用户明确要求“搜索某个方向的论文”时才启用检索流程。

检索时建议：

- 先说明检索主题和关键词
- 先给出少量高相关候选，而不是无控制地堆长列表
- 说明为什么这些论文被纳入
- 检索到的论文仍需回到统一抽取字段

## 质量检查

输出前检查：

1. 是否区分了单篇卡片和跨文献综合
2. 是否所有论文都尽量使用了统一字段
3. 缺失信息是否明确标注
4. 综合部分是否回答了研究脉络、共识、分歧、局限和空白
5. 是否把最终结果组织成一个统一文档
6. 是否在需要时写出了时间演化路径
7. 若是方法比较任务，是否写清了数据链路、表示方式、转换步骤和最终输出
8. 若是 mNGS / taxonomic classification 任务，是否写清了 k 值、表示方式、模型结构、层级任务和数据难度
