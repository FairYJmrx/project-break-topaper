# 难题树构建规则

用于把一个完整项目拆成层层递进的问题结构。

## 核心原则

难题树不是问题清单，而是从流程中长出来的 failure mode 图谱。

每个难题都要回答：

- 出现在流程哪一层？
- 为什么会导致最终目标失败？
- 是数据问题、表示问题、模型问题、决策问题，还是部署问题？
- 是否已有证据显示它存在？
- 它和其他难题是否耦合？

## 标准层级

### 1. 数据层

关注原始数据是否足够、干净、代表真实场景。

常见问题：

- 数据量不足
- 类别不平衡
- 样本偏移
- 噪声和污染
- 标签错误
- 训练/测试泄漏

### 2. 表示层

关注输入如何被编码，信息是否在表示过程中丢失。

常见问题：

- 短输入导致证据不足
- token/k-mer 过粗或过细
- 重要结构信息被平均掉
- 突变或噪声导致表示漂移

### 3. 类别边界层

关注类别之间是否天然难分。

常见问题：

- 近缘类别
- 细粒度类别
- 标签层级接近
- 类别对相似性不均匀
- 类别数量和类别关系复杂

### 4. 模型训练层

关注训练目标是否匹配问题结构。

常见问题：

- 普通交叉熵把所有负类当作等价
- 固定 margin 忽略类别对差异
- long-tail 方法解决的是频率，不是相似性
- 训练增强与测试场景不匹配

### 5. 决策层

关注模型输出如何被解释和纠错。

常见问题：

- 置信度不校准
- top-1 错误但 top-k 有正确答案
- 层级预测不一致
- 不知道何时拒识
- read-level 输出不能直接变成 sample-level 报告

### 6. 部署层

关注真实使用时的外部条件。

常见问题：

- 参考库版本变化
- 未知类别
- 计算成本
- 可复现性
- 临床或领域上下文缺失

## 输出模板

```markdown
## 难题树

### 难题 1：名称

- 流程层级：
- 失败机制：
- 当前证据：
- 与其他难题的关系：
- 可能对策：
- 能否成文：
- 是否能通用化：
```

## 质量检查

- 不要把对策写成难题；
- 不要把未验证假设写成已知事实；
- 不要遗漏部署层问题；
- 对每个难题都要指出“如果不解决，会如何影响最终输出”。
