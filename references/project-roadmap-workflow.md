# 项目级论文路线拆解总流程

用于用户给出一个完整科研项目、技术体系或长期研究方向，并希望判断“有哪些难题、怎么解决、能写几篇论文、能否形成大论文、哪些方案能迁移成通用方法论文”的情况。

## 输出目标

默认输出一份可以持续维护的 Markdown 路线图，包含：

- 项目流程树
- 难题树
- 对策树
- 文献图谱
- 论文边界矩阵
- 阶段化任务面板
- 最终大论文可能性评估

## 主流程

### 1. 项目流程建模

先还原项目从输入到输出的完整链路：

```text
原始输入
  -> 预处理
  -> 表示/特征
  -> 模型或算法
  -> 中间预测
  -> 后处理/纠错
  -> 最终输出
  -> 应用解释
```

要求：

- 不要从单个模块开始；
- 先看整个系统；
- 标出每个环节的输入、输出和可能失败点。

### 2. 构建难题树

读取 `problem-tree.md`。从流程每一层提取困难：

- 数据层困难
- 表示层困难
- 模型层困难
- 决策层困难
- 部署层困难

对用户提出的问题，要判断是否覆盖完整；缺失问题要补充，但不能把未验证问题写成已解决。

### 3. 构建对策树

对每个难题提出候选方案：

- 已有模块
- 可开发模块
- 需要数据或验证的模块
- 暂时不适合做的模块

每个方案必须写清：

- 解决什么问题
- 不解决什么问题
- 需要什么实验
- 成功标准和失败标准

### 4. 构建文献图谱

读取 `solution-literature-map.md`。文献不要按年份机械罗列，而要按“问题-对策-来源领域”组织：

```text
问题
  -> 本领域已有方法
  -> 跨领域方法来源
  -> 与当前方案的关系
  -> 对论文创新的支撑或挑战
```

如果用户没有要求搜索，优先使用已给材料和已知文献；如果用户要求查文献，提醒调用 `nature-academic-search`。

### 5. 划分论文边界

读取 `paper-boundary-matrix.md`。判断每个方案属于：

- 独立论文
- 系统论文主模块
- supplementary / 消融实验
- 后续储备
- 通用方法论文候选

每篇潜在论文都要写：

- 论文问题
- 方法主角
- 核心证据
- 不解决什么
- 目标期刊/会议
- 与其他论文的边界

### 6. 拆成阶段化任务

读取 `task-panel.md`。输出可执行阶段：

- 阶段目标
- 输入
- 任务
- 输出文件
- go/no-go 标准
- 可能新增的论文机会

## 质量检查

输出前检查：

1. 是否先看完整项目，而不是只看一个模块？
2. 是否补充了用户没想到但真实存在的困难？
3. 每个对策是否对应一个具体难题？
4. 是否区分“已验证”“待验证”“未来目标”？
5. 是否给出论文边界，而不是只说可以发论文？
6. 是否指出哪些方案能迁移为通用计算机/方法论文？
7. 是否说明最终大论文能否成立以及何时成立？
8. 是否中文为主，英文术语有解释？
