# 论文边界矩阵规则

用于判断一个项目中的多个方案分别能否成文，以及应该写成什么论文。

## 成文判断标准

一个方案能独立成文，通常需要同时满足：

1. 明确且重要的问题；
2. 可命名的方法或模块；
3. 公平 baseline；
4. 明确指标；
5. 消融或机制证据；
6. 失败边界；
7. 足够清楚的目标读者和投稿方向。

## 五类归属

### 1. 独立论文

适合：

- 问题清晰；
- 方法主角明确；
- 证据闭环完整；
- 与其他模块边界清楚。

### 2. 系统论文主模块

适合：

- 对整体系统很重要；
- 单独创新不够强；
- 但能支撑大论文。

### 3. supplementary / 消融实验

适合：

- 有性能影响；
- 但机制弱或贡献太小；
- 适合作为主方法的支撑证据。

### 4. 后续储备

适合：

- 概念重要；
- 目前数据、验证或方法还不成熟。

### 5. 通用方法论文候选

适合：

- 问题不局限于当前领域；
- 可迁移到非本领域任务；
- 有通用公式、算法或理论解释；
- 能补充跨领域实验。

## 每篇潜在论文必须写清

```markdown
### 论文名称

- 核心问题：
- 方法主角：
- 主要证据：
- 目标读者：
- 目标期刊/会议：
- 不解决什么：
- 与其他论文的边界：
- 是否能通用化：
- 当前成熟度：
- 下一步最小实验：
```

## 大论文判断

判断整个项目能否合成大论文时，问：

1. 是否有一个统一科学目标？
2. 多个模块是否解决同一主线的不同层级？
3. 模块之间是否共享数据流和输出接口？
4. 是否至少有 2-3 个成熟模块？
5. 是否有完整 benchmark 和可复现 pipeline？
6. 大论文是否会稀释最强单点创新？

如果最强模块已经能独立成文，优先先写小而尖的论文，再后期整合大论文。
